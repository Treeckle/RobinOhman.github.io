Warmer Together

April - June, 2025
9 weeks
Unreal Engine, Jetbrains Rider
Group size: 9

Warmer Together is a 2 player co-op where players need to work together in order to climb a mountain.

The main focus of the game is on teamwork, but it also features platforming and puzzles.

My contributions to the project were primarily game mechanics, but I was also responsible for getting the animations into the game.

My contributions

Lifting

Simple logic for lifting

Player 1 holding Player 2

My primary work was based around the usage of physics within Unreal Engine.
Our game features two mechanics that that make use the existing PhysicsHandle component within Unreal Engine. The first mechanic is the ability to lift the other player. While it was originally possible to lift other objects as well, due to player habits of wanting to carry objects with them to other parts of the level, it was reworked to only allow for the smaller character to be picked up for now.

This class is a subclass of a different class I created, which simply holds the logic for grabbing an object using a PhysicsHandle and releasing the object. It also features two empty virtual methods that are called within the respective Grab and Release functions, which are used in the child classes to trigger effects upon performing respective action.

Lifting works by using the PhysicsHandles built in function, GrabComponentAtLocation in combination with a trace that checks if the object is in front of the player. When the PhysicsHandle has grabbed the object, it can easily be manipulated and moved around. While the PhysicsHandle technically wouldn't be required at all when only interacting with the other player character, it is necessary for being able to launch actors that do not inherit from the Character class. The reason why the ability to launch the object is necessary is because of the throwing mechanic, which allows the player to throw its held object.

Pushing

Simple logic for interacting with object

Logic held by pushable objects

The second mechanic that makes use of the PhysicsHandle is the pushing mechanic.
While it's possible to push around objects within Unreal Engine by default through physics collisions, our game desired for it to be more restrictive in nature, which is why I made the pushing component.

It's a subclass of GrabComponent, and uses its logic for grabbing and releasing.
Most of the action is performed within the TickComponent, where all checks are performed to see if the object can be pushed, before updating the objects location. And thanks to the PhysicsHandle, the object can easily me moved using SetTargetLocation, which smoothly moves it without violating collisions.
MaintainRelativeLocation ensures that the player isn't too far away, while UpdateGrabLocation refreshes where the component is grabbed to prevent it from snapping when one player lets go of it when both are holding it. The reason it needs to be updated is that the grabbed object calculates the normal between all of it's grabbers in order to provide said grabbers with the same position, in order to prevent each players own PhysicsHandle from having to fight with each other.

This mechanic could most likely also have been made without using a PhysicsHandle, which also would have most likely simplified some of the logic when it came to multiple players grabbing the same object. I plan on looking into this and experimenting with it in the future.

Minigame

Minigame gameplay

Another mechanic I made was the minigame that is played near the end of the game.
It's a cooperative mechanic meant to force the players to communicate with each other in order to pass it, which it does by having them press the buttons at the same time.
Input is based around a timer, that resets the input when the input window has passed.

Minigame loop

Each player has their own instance of the game, which is managed by another actor inside of the game. This actor keeps track of the state of both minigame instances, and keeps track of the score that has been achieved. When both players have pressed the correct button within the timeframe, one point is awarded and the requested inputs are randomly updated.
This manager also ensures that both players quit out of the minigame correctly, before triggering the cutscene at the end. While it's only capable of triggering the victory cutscene at the moment, it's fully possible to implement an interface that it can call instead on any desired object, to trigger an event when the minigame is completed.

One flaw of the current minigame is that it's possible to mash the button to completely ignore having to time it with the other player, since the timing will likely match up since there is no cooldown between presses. I've been thinking about ways to work around this, which include switching the button every time a press is incorrect, or introducing a cooldown between pressing the button. I personally feel that these solutions are lackluster however.
I would like something that should still be fun, which is why I will have to continue to iterate on it.

Customization

Logic of the customization

I created the customization component for usage in the customization menu, due to the one responsible for the menu not having the time to do it. It can be placed on any actor that possesses a SkeletonMeshComponent, and allows for the texture to easily be swapped with any other texture that has been placed within the component on the blueprint.
It has one function to set to a specific texture right away, and two other functions to iterate through the textures within the array.

It lacks any failsafes due to the fact that it's only intended for usage on 4 specific blueprints, but can be easily implemented if required.

Animation Blueprints

Statemachine belonging to Player 1

I was given the responsiblity of implementing the animations into the game. To this end, I created animation blueprints and state machines in order make it work.
All states are connected to their own booelans and calculations, and most of them should smoothly transition between the different available states.

I set up the walking animations using BlendSpaces, to smoothly transition between walking and running, although the values need to be tweaked after the movement speed of the characters were reworked. I also needed to create new animation through the use of ones we got from Mixamo in addition to the ones we created using motion capture. The reason for this was that some of the animations just didn't look good within in game, and since we didn't have a dedicated animator, I resorted to cutting apart animations in order to create better transitions.
The jump was originally one animation with both jumping and landing, which I cut out most parts off, in order to create a short jump and landing animation that worked with the speed of the characters.
Lifting and Throwing also works through the usage of the Blend by Bone node, which combines the motion captured lift and throw animations with the walking animation from Mixamo.

Other things

Pushable log

Balance log

I've also created or done a lot of smaller things within the game that I don't think require their own section, which is why I will just go over them here.

Pushable Ice Block: The ice blocks that exist within the level that are capable of being pushed. They contain logic for checking if they're currently in the air.

Pushable Log: The logs that can be pushed over. They are controlled using a PhysicsContraint in order to keep them aligned with their intended path while maintaining the smooth physicsbased movement. They contain logic for seeing if it is currently falling, and also a method to check if its rotation has changed between ticks.

Balance Log: A log on top of a rock kept in place through the use of a PhysicsConstraint. It uses the angular motor of the constraint to reset itself to avoid blocking progression.

Music System: A system for easily changing Music between levels through the use of triggers, and a manager that fades in and out the tracks. Also manages the music that plays when a player is dancing.

Interaction System: Wasn't implemented at the time of the video being recorded, but I added a few more functions to the interface that manages the popups to lower the amount of functions called at once from the same button. Pushing now uses the X button together with the hug and climb, and the methods are only called if facing the object the mechanic can be used on.

Debugging: Several small changes within other members code. Includes fixing if statements to prevent triggering multiple actions at once, preventing crashes and optimization.
